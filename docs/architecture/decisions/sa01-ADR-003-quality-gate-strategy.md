# ADR-003: 质量门禁策略

**日期**：2026-04-19
**状态**：accepted
**决策者**：OpenClaw（sa-01 子Agent）

## 上下文（Context）

Claw Design 要求交付前 100% 形成明确结论（NFR-01），假交付率 = 0。质量门禁需要覆盖结构、内容、视觉、品牌和导出五个维度，同时满足首轮结论 ≤ 10 秒的性能要求（NFR-02）。

核心矛盾：检查越深入越准确，但延迟越高。LLM 语义审查最准但成本最高（额外 API 调用），纯规则检查最快但覆盖面有限。

## 备选方案（Options Considered）

### 方案 A：三层分级架构（规则引擎 / DOM 检查 / 可选 LLM）

- 描述：L1 静态规则（< 2s）→ L2 DOM 解析（< 5s）→ L3 LLM 语义（< 15s，默认关闭）
- 优点：分层递进，快速检查覆盖高频问题，深度检查按需启用，总延迟可控
- 缺点：三层维护成本高于单层，L1/L2 边界需要明确划分
- 成本/复杂度：中等

### 方案 B：纯 LLM 审查

- 描述：所有质量检查交给 LLM 做语义判断
- 优点：实现简单，覆盖面广，能理解上下文
- 缺点：每次检查都要 API 调用，延迟 10-30s，成本不可控，与零外部依赖精神冲突
- 成本/复杂度：实现低，运行成本高

### 方案 C：纯规则引擎

- 描述：所有检查用正则、JSON Schema、DOM 查询实现
- 优点：零外部依赖，延迟最低，结果确定性最高
- 缺点：无法覆盖语义层面的质量问题（如"内容是否回应了用户意图"）
- 成本/复杂度：规则维护量大

## 决策（Decision）

选择方案 A：三层分级架构。

L1（规则引擎）和 L2（DOM 检查）默认启用，覆盖结构完整性、内容质量、视觉版式和导出可用性。L3（LLM 语义）默认关闭，用户可通过配置显式开启（FR-D08）。

分层职责：
- L1：页面计数、标题存在性、占位内容残留、数据一致性、格式合规 → Python3 脚本
- L2：字号/对比度/留白/对齐/密度/层级 → cheerio（Node.js）DOM 解析
- L3：意图符合度、信息完整度、逻辑连贯性 → 独立 LLM 调用

## 后果（Consequences）

### 正面
- L1+L2 总延迟 < 7s，满足 ≤ 10s 目标
- 阻断项在 L1/L2 就能拦住，不需要等 LLM
- L3 可选设计尊重用户成本控制权

### 负面
- 三层检查脚本需要分别维护，技术栈跨 Python 和 Node.js
- L1/L2 规则需要持续补充，初期覆盖面有限

### 风险
- L1/L2 规则不足导致低质量产物漏过门禁
- 缓解：质量回路机制（进化机制之一），根据失败案例持续补充规则和反模式清单
