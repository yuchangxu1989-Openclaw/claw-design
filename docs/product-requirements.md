# Claw Design — 产品需求规格说明书
## 产品愿景
Claw Design —— AI 设计引擎，一句话生成演示文稿、图表、架构图、海报等 10 余种设计产物，Claude Design 的免费平替。

它面向真实交付场景：用户说一句话，系统就把需求路由到合适的设计链路，产出可演示、可编辑、可交付、可复用的结果。主打的不是单张图片，而是一整套面向汇报、传播、归档和二次编辑的设计生产能力。

Claw Design 基于 Claude Design 泄露的系统提示词做逆向分析，拆解其意图理解、设计链路组织和结果稳定性机制，独立复现为一套可开源、可扩展、可商用的设计引擎。核心设计方法论全部复现：意图路由、技能管线、质量门禁、主题管理、多格式导出、模板系统。`fork_verifier` 用可选语义审查覆盖（FR-D08），Tweaks 用自然语言增量修订覆盖（FR-B12），`mentioned-element` 用基于 HTML DOM 的元素定位覆盖。唯一不复现的是 Artifacts 渲染沙箱，用 zip 打包 + 浏览器打开替代。

设计硬约束：
- 路由优先：先理解任务，再选能力，再生成。
- HTML 为主表达：先结构化页面，再派生其他格式，HTML + PPTX 双格式输出。
- 零模态依赖：文本 LLM 是唯一必要的 AI 依赖，图像生成、视频生成、视频理解等模态模型均为可选增强项。所有设计 Skill 的渲染层基于 HTML/CSS/SVG/JS，不调用任何模态工具，用户只需配置文本 LLM 即可使用全部设计能力。代码渲染 + 版式系统 + SVG 保证成本稳定、结果可控。
- 预置模板 + 质量门禁：开箱即用的版式方案，交付前自动检查结构、内容、版式和导出。
- 独立项目边界：可独立发布（`clawhub install claw-design`），也可接入上层平台。
### 用户角色
核心用户是内容创作者：产品负责人、咨询顾问、销售、市场运营、设计资源有限的创业团队。扩展角色包括 Agent 开发者、Skill 贡献者、设计系统维护者。
### 关键使用场景
- 把方案文本、会议纪要或提案内容转成演示文稿，并同步得到可编辑版本。
- 把经营、运营或项目数据转成图表、信息图和汇报页。
- 把系统说明转成架构图、流程图、时序图和逻辑关系图。
- 把活动信息转成海报、宣传图和单页展示页。
- 把产品构想转成 UI mockup、线框图和仪表盘页面。
- 把同一主题内容打包成一套配套交付物，用于汇报、传播、归档和二次编辑。
### 成功指标
- 路由准确率 ≥ 90%。
- 质量门禁阻断覆盖率 ≥ 85%。
- 用户无需手动拆解任务或指定工具，一句话触发完整流程。
### 交付物质量标准
- 内容结构完整，主辅信息关系清楚。
- 视觉风格一致，颜色、字体、间距、图形语言前后一致。
- 结果可编辑、可交付、可读、可复现、可校验。
## 概念架构
### 对象类型
- Design Request：用户原始需求，含目标、场景、附件和限制条件。
- Intent Packet：识别和收束后的任务意图包，定义主产物、附属产物、优先级、缺口和必要上下文。
- Theme Pack：任务级视觉上下文容器，承接品牌规则、风格倾向、版式偏好、历史风格和可复用模块。
- Brand Kit：Theme Pack 的结构化子集，聚焦颜色、字体、标识、间距、版式规则等品牌约束。
- Artifact Spec：交付物结构定义，描述页面数、信息层级、模块清单、主视觉重点和产物关系。
- Skill：生成特定交付物的能力单元，通过统一合同声明输入、输出、上下文需求和质量预期。
- Artifact：设计产物本身，含主结果和元数据。生命周期状态：
  - generating：Skill 正在生成。
  - ready：生成完成，可供质量检查、导出和交付。
  - revision：用户发起修订，正在增量修改。
  - archived：交付完成或主动归档。
  流转：generating → ready ⇄ revision → archived。质量门禁在 ready 状态执行并给出交付结论，交付打包在质量通过后进行，增量修订将 ready 产物转入 revision、完成后回到 ready。
- Export Adapter：把主结果派生成其他交付格式的适配能力。
- Quality Report：质量门禁结果，记录通过项、提醒项、阻断项和人工建议。
- Delivery Bundle：最终交付包，含产物文件、用途分组、阅读说明和质量摘要。
### 管线流转
```text
Design Request
  → Intent Packet
  → Theme Pack 收束
  → Artifact Spec 组装
  → Skill 生成 Artifact
  → Quality Gate 校验
  → Export Adapters 派生格式
  → Delivery Bundle 交付
```
七段职责：
1. 输入接收：收集自然语言需求、附件、品牌素材、历史稿件、结构化数据和参考样式。
2. 意图判定：识别主交付物、附属交付物、复杂度和待补信息。
3. 上下文收束：形成 Intent Packet 和 Theme Pack，明确内容边界与风格边界。
4. 结构成稿：拆成一个或多个 Artifact Spec，约定信息层级和产物关系。
5. 技能生成：按交付物类型调用对应 Skill，生成结构化设计产物。
6. 质量门禁：检查结构、内容、视觉、品牌和导出可用性。
7. 交付打包：按用途组织交付物，附质量摘要和修改入口。
### 系统边界
- Claw Design 管理从任务理解到交付打包的完整链路。
- Claw Design 负责识别缺口并生成结构化澄清项，宿主负责向用户发问和回填答案。
- Claw Design 负责输出交付包和结果摘要，宿主负责把结果送到具体渠道。
- Claw Design 不把主要能力建立在外部图片生成服务上。
- Claw Design 不绑定固定宿主环境，但允许被各类 Agent 和工作流平台调用。
### 进化机制
- 路由回路：根据误判、纠偏和缺口补齐记录提升意图识别质量。
- 模板回路：根据高频任务和人工修订方向沉淀模板骨架和变体。
- 质量回路：根据失败案例和人工反馈补充门禁规则与反模式清单。
- 生态回路：根据行业场景和社区贡献扩展 Skill、主题包和联产方案。
## 一、功能需求（FR）
### 域 A：意图路由（Intent Router）
负责判断用户要什么、缺什么、该走哪条链路。
#### FR-A01：自然语言意图识别
识别自然语言、附件和上下文里的主交付物类型。
- AC1：系统至少支持识别以下十类主交付物：演示文稿、图表、架构图、流程图或时序图、海报、Landing Page、UI mockup 或线框图、Dashboard、信息图、逻辑关系图。
- AC2：当请求同时包含多个目标时，系统能区分主交付物和附属交付物。
- AC3：当意图置信度不足时，系统返回候选类型和澄清建议，不直接进入错误生成链路。
- AC4：当用户输入已包含明确的产物类型和内容主题且无歧义时，系统跳过澄清直接进入生成。判断条件：IntentClassifier 返回单一候选且置信度达到阈值、输入中包含可提取的实质性内容主题（≥ 1 个实体或主题词，非纯格式指令）、且 GapDetector 缺口清单为空或仅包含 SHOULD 级约束的可选项。产物类型判断由 IntentRouter 输出，Clarify 只消费该结果做跳过判定，不自行推断产物类型或检查 Skill 注册状态。
#### FR-A02：需求缺口澄清
输入缺少关键上下文时，生成聚焦的澄清项。
- AC1：系统至少能识别受众、用途、风格、篇幅、品牌约束、信息来源、输出格式和时间限制等缺口。
- AC2：澄清结果以结构化清单输出，包含缺口名称、缺口原因、优先级和可接受的默认值范围。
- AC3：当任务要求快速出稿时，系统可在清单中标记哪些缺口允许先用默认骨架继续推进。
#### FR-A03：上下文包识别
识别相关上下文包，并判断影响环节。
- AC1：系统能识别品牌规范、已有稿件、页面截图、历史物料、参考样式和结构化数据等输入材料。
- AC2：系统能区分可复用上下文和当前任务专属上下文。
- AC3：当上下文之间存在冲突时，系统输出冲突点和优先级建议。
#### FR-A04：Skill 路由选择
基于 Skill 能力声明合同（FR-F01）选择可执行能力。
- AC1：路由层读取每个 Skill 的能力声明合同，根据产物类型、输入匹配度和上下文满足度选择最合适的 Skill。
- AC2：路由层只在满足合同条件的 Skill 范围内做选择。
- AC3：新增 Skill 按统一合同声明能力后自动进入路由体系，不需要改写已有路由规则。
#### FR-A05：多意图组合路由
支持一条请求触发多个设计意图，并保留依赖关系。
- AC1：系统可识别“先做演示文稿，再配套图表”“先做架构图，再嵌入汇报页”这类组合任务。
- AC2：组合结果输出时包含产物依赖关系和执行顺序说明。
- AC3：嵌入型意图不会破坏主交付物的主题和信息主线。
#### FR-A06：多产物联产编排
支持同一任务生成一组配套交付物。
- AC1：用户可一次性获得主交付物和附属传播物料。
- AC2：联产结果共享同一主题上下文、同一信息主线和同一品牌约束。
- AC3：系统输出时标明每个产物的用途、推荐使用场景和建议使用顺序。
#### FR-A07：路由结果可解释
输出用户能看懂的路由结论。
- AC1：路由结果至少包含主交付物、附属交付物、关键缺口、已采纳上下文和计划触发的质量门禁类型。
- AC2：业务用户能读懂本次任务为什么走这条链路。
- AC3：当系统判断偏差时，用户或宿主可基于路由说明触发重新路由。
#### FR-A08：路由可观测性
为每次路由和后续结果保留可复盘记录。
- AC1：系统能关联记录路由结论、澄清结果、生成结果、质量状态和交付状态。
- AC2：复盘时能区分路由误判、上下文缺失、模板不匹配、导出问题等不同失败来源。
- AC3：同类任务的纠偏结果可用于后续路由优化。
#### FR-A09：主动交互式需求澄清
在设计链路全阶段检测意图模糊或决策缺口时，触发结构化提问，收敛后才推进下一阶段。
- AC1：routing 阶段检测到用户意图模糊时，系统针对设计领域关键决策因素生成结构化提问清单，覆盖目标受众、风格偏好、品牌调性、内容范围、交付格式、色彩倾向、信息密度和使用场景。
- AC2：execution 阶段在设计方案生成前，系统对关键设计决策（版式选择、信息层级、视觉重点）发起确认式提问，用户确认或修正后才继续生成。
- AC3：quality 阶段在交付前，系统对存疑项（如品牌合规不确定、内容取舍有歧义）发起最终确认，用户回应后才进入交付打包。
- AC4：提问以工具化接口输出（参考 questions_v2 机制），每次提问包含问题文本、决策因素类型、可选默认值和跳过条件，宿主平台消费该接口向用户发问并回填答案。
- AC5：澄清收敛判定条件：所有 MUST 级缺口已回填、或用户显式确认采用默认值、或用户发出"直接生成"指令。收敛前系统不进入下一阶段。
- AC6：单阶段最多提问 2 轮，超过后对剩余未回填缺口采用默认值继续推进，避免过度追问用户。
- AC7：实现路径为 skill 层提供澄清模板和决策因素词表，通过 prompt 注入方式将澄清规则植入各阶段执行上下文，不硬编码到路由逻辑中。
### 域 B：设计技能（Design Skills）
每类交付物是一个独立 Skill，统一接入主题、质量门禁和导出规则。
#### FR-B01：演示文稿生成
支持基于文本生成演示文稿，并输出可演示结果和可编辑结果。
- AC1：支持标题页、目录页、正文页、对比页、图表页、总结页等常见页面类型。
- AC2：主结果支持稳定翻页和连续浏览。
- AC3：导出结果保留可编辑文本和主要版式关系。
#### FR-B02：图表与数据可视化生成
支持把表格、列表或自然语言中的数据转成图表与可视化组件。
- AC1：支持柱状图、折线图、饼图、漏斗图、桑基图、热力图、对比卡片等常见类型。
- AC2：图表包含标题、图例、数值标签或必要说明。
- AC3：主结果与导出结果保持核心信息一致。
#### FR-B03：架构图与系统图生成
支持生成架构图、系统边界图、组件关系图和部署关系图。
- AC1：图中节点、分组、连接关系和层级关系清晰可读。
- AC2：支持浅色和深色两种展示主题。
- AC3：关键图形优先保留矢量表达能力，便于复用和二次编辑。
#### FR-B04：流程图与时序图生成
支持把流程说明、业务规则和交互步骤表达为流程图、泳道图和时序图。
- AC1：流程开始、决策、分支、收束和参与者顺序表达清楚。
- AC2：复杂流程能按模块拆段，避免整图拥挤。
- AC3：文字说明与连线逻辑保持一致。
#### FR-B05：海报与宣传图生成
支持生成纯代码驱动的海报与宣传图。
- AC1：海报在不依赖外部图片生成服务的情况下也能完成主视觉构图。
- AC2：支持文字层级、背景层、强调元素、装饰图形和品牌标识编排。
- AC3：结果可适配常见宣传尺寸和后续导出需求。
#### FR-B06：Landing Page 生成
支持生成单页式展示页，用于产品介绍、活动说明和方案展示。
- AC1：页面包含首屏、价值说明、核心卖点、内容分区和行动入口。
- AC2：页面风格与主题包保持一致。
- AC3：页面结果可直接分享和继续改版。
#### FR-B07：UI mockup 与线框图生成
支持生成产品界面 mockup、线框图和概念页面。
- AC1：支持桌面端和移动端两类常见画布。
- AC2：支持低保真和高保真两种表达层级。
- AC3：界面组件具备清楚的信息层级和状态提示。
#### FR-B08：Dashboard 仪表盘生成
支持把经营、运营、产品和项目数据组织成 Dashboard 页面。
- AC1：支持指标卡、趋势图、明细区、筛选区和状态区组合。
- AC2：信息密度和阅读节奏适合汇报场景与日常查看场景。
- AC3：同一份数据可生成演示型和管理型两类版式。
#### FR-B09：信息图生成
支持把复杂内容压缩为单页信息图。
- AC1：信息图能表达层级、步骤、对比、数字亮点和结论。
- AC2：版面可纵向或横向展开，适应不同传播场景。
- AC3：缩略查看时仍保留主信息识别度。
#### FR-B10：逻辑关系图生成
支持把长文本、论证结构和概念体系拆解为逻辑关系图。
- AC1：支持递进、并列、对比、循环、层级和矩阵等逻辑关系表达。
- AC2：图形表达与原文逻辑对应，不改变原意。
- AC3：复杂内容可拆成多张图连续表达。
#### FR-B11：Skill 共享能力底座
为所有 Design Skill 提供共享能力底座。
- AC1：共享能力至少覆盖主题变量、图形与图标基元、版式骨架和反模式清单。
- AC2：不同 Skill 生成的结果能复用同一套视觉规则和基础模块。
- AC3：共享能力更新后，新任务可直接继承改进效果。
#### FR-B12：结果修订与增量调整
支持基于已有 Artifact 做增量修订和持续收敛。
- AC1：用户可对已生成结果提出修改要求，包括调字号、换配色、改内容、增删页面等常见修订动作。
- AC2：系统在原有结果基础上做增量修改，不因局部修订而默认从零重新生成整个交付物。修订操作将 Artifact 从 ready 状态转入 revision，完成后回到 ready。
- AC3：修订历史可追溯，用户可回退到之前的版本继续调整。
#### FR-B13：交互原型生成
支持生成可点击的高保真 HTML 交互原型，用于产品构想验证和交互逻辑演示。该能力面向可点击、有状态切换、有页面路由的功能验证；FR-B07 面向静态视觉稿和不可交互的 UI mockup。路由层根据用户意图中是否包含“可点击”“交互”“跳转”“状态切换”等信号判断进入 FR-B13 链路，否则按静态界面表达进入 FR-B07。
- AC1：系统能生成包含可点击 UI 组件（包括但不限于按钮、表单、卡片、导航栏、弹窗等）的 HTML 原型，组件带 JS 事件绑定，支持点击反馈、状态切换和表单交互。
- AC2：系统支持单文件多页面原型，通过 hash router 实现页面间导航，用户可点击链接或按钮在页面之间跳转。
- AC3：系统支持基于 data-state 属性的状态管理，组件可根据用户操作切换显示状态（如 tab 切换、折叠展开、弹窗显隐）。
- AC4：原型生成由 prototype skill 提供交互骨架模板（路由模板、组件模板、状态管理模板），LLM 负责填充具体内容、布局和业务逻辑。
- AC5：原型结果为独立 HTML 文件，可直接在浏览器中打开和交互，无需构建工具或运行时依赖。
- AC6：原型支持桌面端和移动端两种视口适配，通过响应式布局或独立画布切换。
### 域 C：导出适配器（Export Adapters）
从同一份主结果派生不同交付格式。
#### FR-C01：独立 HTML 交付
支持把主结果交付为可独立打开和分享的 HTML。
- AC1：HTML 结果可在常见浏览环境中直接打开。
- AC2：交付物以 zip 压缩包形式提供，包含完整的依赖资源，无需网络连接即可浏览，并支持后续增量修改。
- AC3：关键图形优先保留结构化表达。
#### FR-C02：PPTX 可编辑导出
支持从同一任务结果中同步得到可编辑 PPTX。
- AC1：PPTX 保留可编辑文本、块级结构和主要图形关系。
- AC2：导出后的文件能被常见办公工具打开。
- AC3：用户可在导出结果上继续做文字和版式微调。
#### FR-C03：PPTX 回退导出
页面结构过于复杂时，支持必要的回退导出策略。
- AC1：系统能识别哪些页面难以完整保留可编辑结构。
- AC2：回退策略只影响必要页面，不拖累整个交付物。
- AC3：导出说明中会标明发生回退的页面和原因。
#### FR-C04：PDF 导出
支持把交付物输出为适合归档和打印的 PDF。
- AC1：页序、标题层级和主要视觉内容在 PDF 中保持稳定。
- AC2：结果适合归档、打印和正式发送。
- AC3：同一任务的 PDF 与主结果保持信息一致。
#### FR-C05：PNG / SVG 导出
支持把单页或单图结果导出为图片或矢量文件。
- AC1：图表、架构图、海报和单页图解支持单独导出。
- AC2：当产物具备矢量表达条件时，优先保留矢量可复用能力。
- AC3：导出结果适合嵌入文档、社交传播和二次设计。
#### FR-C06：跨格式一致性
保证不同格式在信息结构和主视觉语言上保持一致。
- AC1：标题、页序、核心图表和重点结论在主要格式之间能一一对应。
- AC2：不同格式允许局部适配，但不能改变核心表达。
- AC3：当格式差异不可避免时，系统给出差异说明。
#### FR-C07：交付打包
支持把任务产出组织为可分发、可归档、可继续协作的交付包。
- AC1：交付包至少包含产物文件、产物清单和阅读说明。
- AC2：多产物任务按用途分组，便于业务用户直接使用。
- AC3：交付包附带质量摘要和后续可调整方向。
- AC4：HTML 类交付物及其依赖资源（CSS、JS、字体、SVG、图片）打包为单个 zip 压缩包，zip 内包含 `index.html` 和 `assets/` 目录，解压后可直接在浏览器中打开。
### 域 D：质量门禁（Quality Gate）
独立于生成链路的验证层，负责给任务下结论。
#### FR-D01：结构完整性检查
交付前检查产物结构是否完整。
- AC1：检查页面缺失、标题缺失、占位内容残留、图表空白、导出页数异常等问题。
- AC2：发现严重结构问题时，任务不能直接进入交付状态。
- AC3：检查结果附带明确的问题说明和问题位置。
#### FR-D02：内容质量检查
检查内容是否完整、清楚且与输入一致。
- AC1：关键区域不能为空白，不能保留未替换占位内容。
- AC2：图表与数据表达要和输入事实一致。
- AC3：当页面文字密度失衡或关键结论缺失时，系统给出明确结论。
#### FR-D03：视觉与版式检查
检查可读性、对齐、密度、对比和层级是否达到交付底线。
- AC1：系统能识别字号过小、对比不足、留白失衡、页面过满、元素拥挤和层级混乱等常见风险。
- AC2：检查结果区分阻断项和提醒项。
- AC3：影响可读性和商业可交付性的严重问题会阻断交付。
#### FR-D04：品牌合规检查
检查产物是否遵循 Theme Pack 和 Brand Kit 约束。
- AC1：系统检查颜色、字体倾向、标识使用和版式节奏是否落在规定边界内。
- AC2：提供了主题包时，结果必须优先服从主题包约束。
- AC3：未提供品牌上下文时，系统采用统一基线风格并在结果中说明。
#### FR-D05：反模式检查
通过反模式清单减少低质量产物。
- AC1：系统能识别常见低质量排版、信息堆砌、风格漂移和模板化痕迹。
- AC2：反模式检查根据影响程度输出阻断项或提醒项。
- AC3：当用户明确要求特殊风格时，系统允许局部豁免，但保留风险提示。
- AC4：反模式清单以结构化数据维护，新增条目只需追加数据记录，不需要修改检查逻辑代码。
- AC5：命中阻断级反模式时，系统同时输出替代建议，说明如何规避该反模式并给出可参考的改进方向。
#### FR-D06：跨格式与导出可用性检查
在导出前后检查不同格式的可用性与对应关系。
- AC1：导出前确认当前产物满足目标格式的基本条件。
- AC2：导出后检查页数、主标题、主要图表和重点结论是否对应。
- AC3：导出失败时，系统返回具体失败项和受影响范围。
#### FR-D07：质量报告与交付结论
为每次任务生成结构化质量报告，并用它决定是否允许交付。
- AC1：质量报告至少包含通过项、提醒项、阻断项和人工建议项。
- AC2：质量结论按产物分组展示，避免多产物任务互相混淆。
- AC3：交付动作以质量结论为依据，存在阻断项时不能进入正式交付。
#### FR-D08：语义交叉验证
可选支持在设计生成完成后调用独立 LLM 做意图符合度审查。
- AC1：该功能默认关闭，用户可通过配置或任务参数显式开启。
- AC2：开启后，系统将用户原始需求和生成结果一起发给独立 LLM，评估结果是否回应了用户意图。
- AC3：验证结果纳入质量报告（FR-D07），作为独立检查项展示。
- AC4：用户知情该功能会增加一次 LLM 调用的成本和延迟。
#### FR-D09：全局 Slop 黑名单（Global Slop Blacklist）
一组与品牌无关的通用 AI 味视觉反模式规则集，作为全局拦截层独立于品牌包运行。不管当前激活什么品牌包，黑名单中的反模式都不允许出现在最终产物中。
- AC1：系统内置至少 20 条 Slop 规则，每条规则包含 id（kebab-case 唯一标识）、name（人类可读名称）、description（反模式描述）、detection（检测逻辑描述，说明如何在 HTML/CSS 产物中识别该模式）、severity（block 或 warn）四个必填字段。
- AC2：内置规则覆盖以下 AI 味反模式（severity=block 除非另注）：
  - `generic-purple-gradient`：紫蓝渐变背景（purple-to-blue / purple-to-pink linear-gradient 作为大面积背景）。
  - `emoji-as-icon`：用通用 emoji 字符代替专业 SVG/icon-font 图标。
  - `inter-as-display`：Inter 字体用作标题或展示字体（Inter 只适合 body text）。
  - `hand-drawn-faces`：手绘风格 SVG 真人脸或通用人物插画。
  - `fabricated-data`：编造统计数字或百分比（没有真实数据源时应使用破折号或标注"示例数据"）。
  - `stock-photo-description`：在 img alt 或注释中使用通用 stock photo 风格描述（如"happy diverse team in modern office"）。
  - `lorem-ipsum`：残留 Lorem ipsum、placeholder text 或明显占位文本。
  - `excessive-drop-shadow`：同一页面超过 3 个元素使用 drop-shadow 且参数相同。
  - `all-pill-buttons`：页面中所有按钮都是全圆角药丸形（border-radius ≥ 999px 或 50%）。
  - `uniform-card-grid`：页面主体内容区仅由相同尺寸的卡片网格组成，无视觉层级变化。
  - `glassmorphism-overuse`：超过 2 个容器同时使用 backdrop-filter: blur + 半透明背景。
  - `generic-gradient-blob`：使用大面积模糊渐变色块（blob）作为纯装饰背景元素。
  - `centered-everything`：页面所有内容块均居中对齐（text-align: center），无左对齐或层级变化。
  - `generic-isometric-illustration`：使用通用等距（isometric）插画风格作为主视觉。
  - `excessive-border-radius`：所有容器元素统一使用 ≥ 16px 大圆角，无圆角层级区分。
  - `dark-neon-combo`：深色背景搭配霓虹色（高饱和度亮色）文字或边框作为主视觉风格。
  - `generic-hero-layout`：hero section 采用"大标题 + 副标题 + 单个 CTA 按钮垂直居中"的千篇一律布局。
  - `gradient-text-overuse`：超过 2 处文字使用 background-clip: text + 渐变色效果。
  - `generic-testimonial-cards`：三列等宽推荐语卡片（头像 + 引言 + 姓名），无设计差异化。
  - `ai-avatar-placeholder`：使用 AI 生成的通用人物头像或圆形纯色占位头像。
- AC3：每条规则的 detection 字段描述具体的 HTML/CSS 检测逻辑，使检查器能基于 DOM 结构和样式属性做自动化判定，不依赖视觉模型。
- AC4：Slop 黑名单与 FR-D05 的通用反模式检查互补但不重叠——FR-D05 覆盖排版质量和信息组织问题，Slop 黑名单专注 AI 生成工具的典型视觉陋习。
- AC5：黑名单规则以结构化 JSON/TS 数据文件维护，与检查逻辑代码分离，新增规则只需追加数据记录。
#### FR-D10：Slop 黑名单配置与管理（Slop Blacklist Configuration）
用户可以自定义 Slop 黑名单：添加自定义规则、禁用内置规则、调整严重级别。
- AC1：提供配置入口（配置文件或 CLI 命令），用户可添加自定义 Slop 规则，格式与内置规则一致（id + name + description + detection + severity）。
- AC2：用户可通过配置禁用特定内置规则（按 id 指定），禁用后该规则不参与检查。
- AC3：用户可将内置规则的 severity 从 block 降级为 warn，或从 warn 升级为 block。
- AC4：自定义规则与内置规则享有完全相同的执行优先级和检查流程。
- AC5：配置变更即时生效，不需要重启或重新初始化。
- AC6：提供 `npx claw-design slop-list` 命令，输出当前生效的全部 Slop 规则（含状态：启用/禁用、来源：内置/自定义、severity）。
- AC7：当用户添加的自定义规则 id 与内置规则冲突时，系统拒绝添加并提示冲突。
#### FR-D11：Slop 检查执行管线（Slop Check Execution Pipeline）
定义 Slop 黑名单检查的执行时机、判定逻辑和强制重做机制。
- AC1：Slop 检查在品牌约束校验（FR-D04 + FR-H08）完成之后、最终交付打包之前执行，确保产物已通过品牌合规但尚未进入不可逆的交付流程。
- AC2：检查器遍历当前生效的全部 Slop 规则（内置 + 自定义，排除已禁用），逐条对生成产物的 HTML/CSS 做模式匹配。
- AC3：命中 severity=block 的规则时，产物被判定为不合格，系统自动触发重新生成（最多 3 次重试），重新生成的 prompt 中注入具体命中的规则描述和规避指导。
- AC4：命中 severity=warn 的规则时，产物不被阻断，但警告信息纳入质量报告（FR-D07），用户可见。
- AC5：单次检查命中多条 block 规则时，全部命中项一次性注入重新生成 prompt，避免逐条重试的低效循环。
- AC6：3 次重试后仍命中 block 规则时，产物标记为质量不达标，进入质量报告的阻断项，附带全部命中规则的 id、描述和检测结果，由用户决定是否接受或手动修改。
- AC7：Slop 检查结果在质量报告（FR-D07）中作为独立检查类别展示，与结构检查、内容检查、视觉检查、品牌检查并列。
- AC8：当用户通过 FR-D05 AC3 的豁免机制明确声明某种风格时（如"我就要霓虹风格"），对应的 Slop 规则自动降级为 warn，不阻断生成。
### 域 E：品牌上下文（Theme Pack / Brand Kit）
管理任务级视觉上下文和品牌约束，为生成链路提供统一风格边界。
#### FR-E01：Theme Pack 模型
支持把视觉上下文抽象为可复用的 Theme Pack。
- AC1：Theme Pack 至少覆盖颜色倾向、字体倾向、间距节奏、版式偏好、图形语言和标识规则。
- AC2：Theme Pack 可同时承接品牌规则、历史风格和任务级风格要求。
- AC3：同一个 Theme Pack 可被多个任务复用。
#### FR-E02：Brand Kit 组装与提取
支持从明确规则或现有素材中组装 Brand Kit。
- AC1：用户可直接提供品牌规范，也可提供已有稿件、截图、页面或历史物料作为来源。
- AC2：系统能从来源材料中提取可复用的品牌线索并形成结构化结果。
- AC3：自动提取出的结果会标记确认状态，便于后续修正。
#### FR-E03：模板与变体
支持同一交付物生成多个真实可选的风格变体。
- AC1：同一内容和主题下，系统至少能生成两种可区分的版式或风格方案。
- AC2：变体差异体现在信息节奏、版式组织和图形语言上，不能只是简单换色。
- AC3：用户选中某一变体后，可继续在该基础上增量调整。
##### 预置模板清单
- 演示文稿（Slides）— 5 套：商务简约、科技深色、学术清新、创意彩色、极简黑白。
- 图表（Charts）— 3 套配色方案：商务蓝、暗色主题、明亮活泼。
- 架构图（Architecture）— 3 套风格：线框简约、深色技术、彩色分层。
- 其他交付物类型：海报、流程图、逻辑图、Landing Page、UI mockup、Dashboard、信息图等，模板待定义。
#### FR-E04：可复用组件资产
支持积累可跨任务复用的组件资产。
- AC1：标题区、图表容器、封面结构、对比模块、流程模块和数据卡片等模块可重复复用。
- AC2：同类资产可跨主题使用，并保留结构稳定性。
- AC3：资产更新后，后续任务可直接继承新版本能力。
#### FR-E05：主题应用与冲突处理
明确 Theme Pack、Brand Kit 和任务材料之间的优先关系。
- AC1：主题上下文会影响模板选择、变体生成、视觉语言和导出策略。
- AC2：当多个来源给出冲突规则时，系统输出优先级判断和采用结果。
- AC3：同一任务产生的所有交付物保持一致的主题边界。
### 域 F：Skill 扩展协议（Skill Extension Protocol）
定义新能力如何接入、如何被发现、如何保持质量一致。
#### FR-F01：能力声明合同
定义所有 Skill 的统一能力声明合同，内置 Skill 和扩展 Skill 共用同一份合同结构。
- AC1：每个 Skill 声明产物类型、适用场景、输入范围、所需上下文、支持输出和质量预期。
- AC2：路由层、质量门禁和导出适配器都能读取同一份能力声明。
- AC3：新能力接入后保持统一入口体验。
#### FR-F02：输入输出接口定义
为扩展 Skill 提供清楚的输入输出接口定义。
- AC1：每个 Skill 都要说明接收什么输入、产出什么主结果、支持哪些附加格式。
- AC2：每个 Skill 都要说明自己需要的 Theme Pack 字段和质量检查覆盖范围。
- AC3：宿主平台能基于接口定义稳定调用 Skill 并消费结果。
#### FR-F03：Skill 生命周期管理
支持 Skill 从接入到退出的完整生命周期。
- AC1：Skill 至少具备注册、测试、发布、更新和退役五类生命周期动作。
- AC2：版本变化不会默默破坏已有任务路径。
- AC3：失效或退役的 Skill 不会继续参与新任务路由。
#### FR-F04：执行边界与隔离
限制扩展 Skill 的执行边界，保证核心链路稳定。
- AC1：扩展 Skill 只能在声明过的输入输出边界内运行。
- AC2：扩展 Skill 不能绕开统一主题、质量和交付规则。
- AC3：单个 Skill 的失败不会拖垮整条产品链路。
#### FR-F05：生态发现与分发入口
提供面向社区和商业扩展的统一发现入口。
- AC1：用户能知道当前有哪些可安装或可启用的扩展能力。
- AC2：扩展能力能按产物类型、行业场景和质量成熟度被组织和筛选。
- AC3：生态扩展不会破坏核心产品边界和交付标准。
### 域 G：设计系统约束层（Design System Constraint Layer）
把设计决策从 LLM 运行时移到模板层。LLM 只做内容编排（选 section、填文案），视觉参数由预定义的设计系统包锁定，消除 LLM 视觉决策能力不足导致的输出质量波动。
#### FR-G01：设计系统包（Design System Package）
定义设计系统包的结构和管理机制。每个设计系统包是一组完整的视觉规则集合，包含 CSS 变量（色彩体系）、间距尺度、字号层级、圆角值、阴影与透明度规则、组件 class 定义。支持多套风格并存（暗色科幻、极简亮色、企业蓝等），每套规则极其具体、极其克制，不给 LLM 任何视觉自由度。
- AC1：设计系统包以标准 CSS 变量 + 预定义 class 的形式存在，可独立加载和切换。
- AC2：给定一个设计系统包，LLM 生成的页面必须 100% 使用包内定义的变量和 class，不允许自定义颜色、间距或字号。
- AC3：系统支持同时注册多套设计系统包，生成时按任务指定或默认规则选择。
#### FR-G02：设计决策与内容决策分离
生成网页时，LLM 的职责严格限定为内容编排：选择 section 组合、决定每个 section 放什么内容、撰写文案。视觉参数（颜色、间距、圆角、阴影、动效）全部由当前激活的设计系统包锁定，LLM 不做任何视觉决策。
- AC1：生成的 HTML 中不允许出现硬编码的颜色值（hex/rgb/hsl）、px 间距值、字号值——必须全部引用 CSS 变量或预定义 class。
- AC2：生成 prompt 中明确注入「禁止自定义视觉参数」的约束指令。
- AC3：违反分离原则的生成结果能被质量门禁（域 D）检出并阻断。
#### FR-G03：参考驱动生成（Reference-Driven Generation）
每次生成必须基于一个参考页面。参考页面是设计系统包的实例化样本，展示该包在真实内容下的视觉效果。LLM 的任务是在这个视觉框架内填充新内容，而非从零构建视觉。
- AC1：每个设计系统包至少附带一个参考页面作为生成基准。
- AC2：生成结果与参考页面的视觉一致性评分 ≥ 90%（通过截图对比验证）。
- AC3：参考页面可由用户指定或由系统根据任务类型自动匹配。
#### FR-G04：质量约束注入（Constraint Injection）
生成时自动向 LLM prompt 注入强制约束规则和禁止规则，从源头限制输出偏离设计系统。
- AC1：强制规则：色彩数 ≤ 设计系统包声明的色板数量、字号层级 ≤ 包声明的层级数、间距值必须来自包定义的固定尺度。
- AC2：禁止规则：禁止 LLM 自行定义颜色值、禁止自行决定 padding/margin 数值、禁止引入包外字体。
- AC3：违反约束的生成结果自动被拒绝并触发重新生成（最多 3 次重试）。
#### FR-G05：视觉验证闭环（Visual Verification Loop）
生成完成后自动执行视觉验证，将生成结果与参考页面做截图对比，偏离超过阈值则自动重新生成。
- AC1：生成后自动截图并与参考页面做像素级或结构级视觉对比。
- AC2：偏离超过可配置阈值（默认 10%）时自动触发重新生成，最多重试 3 次。
- AC3：最终交付的页面必须通过视觉一致性检查，未通过则标记为质量不达标并报告具体偏离项。
### 域 H：品牌包生态（Brand Package Ecosystem）
把设计系统包从单一内置扩展为可规模化的品牌包生态。每个品牌包采用双层架构：Markdown 描述层传递设计意图和氛围（注入 LLM prompt），TypeScript 结构层锁定硬参数（驱动 CSS 生成和约束校验）。目标是让用户说出品牌名即可获得该品牌风格的设计产物，同时保证输出质量可控。
#### FR-H01：品牌包双层格式（Dual-Layer Brand Package）
每个品牌包同时包含 Markdown 描述文件和 TypeScript 结构化模块，两层各司其职。
- AC1：每个品牌包目录包含 `design.md`（设计意图描述）、`index.ts`（结构化参数，export DesignSystemPackage）和 `reference.html`（视觉基准页面）三个文件。
- AC2：`design.md` 遵循 9 段标准结构（Visual Theme → Color Palette → Typography → Component Stylings → Layout Principles → Imagery → Animation → Accessibility → Brand Voice），每段至少包含一个可提取的硬参数值。
- AC3：`index.ts` 的 export 类型必须与 Domain G 定义的 `DesignSystemPackage` 接口完全一致，包含 tokens、components、reference、constraints 四个顶层字段。
- AC4：`design.md` 中的自然语言描述和 `index.ts` 中的结构化参数不矛盾——颜色 hex 值、字体名、间距数值在两层中保持一致。
- AC5：缺少任一必要文件（`design.md` 或 `index.ts`）的目录不被识别为有效品牌包。
#### FR-H02：MD→TS 批量转换器（Design Markdown Converter）
提供 LLM 辅助的转换脚本，从 DESIGN.md 提取硬参数生成 TypeScript 模块，支持单个和批量转换。
- AC1：转换器以 CLI 脚本形式存在（`scripts/convert-design-md.ts`），接受单个 DESIGN.md 路径或包含多个 DESIGN.md 的目录路径作为输入。
- AC2：转换器从 DESIGN.md 的 §2（Color Palette）提取颜色 hex 值生成 `tokens.colors`，从 §3（Typography）提取字体族和尺寸生成 `tokens.typography`，从 §4（Component Stylings）提取 CSS 属性生成 `components[]`，从 §5（Layout Principles）提取间距和圆角生成 `tokens.spacing` 和 `tokens.radius`。
- AC3：转换器为每个品牌包自动生成通用约束规则（颜色必须来自 tokens、字体必须来自声明、间距必须来自阶梯），约束规则以可执行函数形式写入 `constraints` 字段。
- AC4：批量模式下，转换器逐个处理目录中的 DESIGN.md，单个转换失败不阻断其余文件，失败项记录到错误日志并在最终输出汇总报告。
- AC5：转换器产出的 TypeScript 文件通过 `tsc --noEmit` 类型检查，无编译错误。
- AC6：转换器是开发时一次性工具，不作为运行时依赖打包到最终产物中。
#### FR-H03：品牌包注册表（Package Registry）
统一管理所有已安装的品牌包，提供检索、列举和元数据查询能力。
- AC1：注册表自动扫描 `src/design-system/packages/` 目录下所有有效品牌包（含 `design.md` + `index.ts`），无需手动注册。
- AC2：注册表提供按名称精确匹配、按标签筛选（如 fintech、consumer、saas）和按风格关键词模糊搜索三种检索方式。
- AC3：注册表为每个包暴露元数据：id、name、version、description、tags、色板预览（前 5 个主色 hex）和可用状态。
- AC4：注册表在系统启动时完成扫描，运行时通过内存缓存提供查询，新增包后支持热重载（无需重启）。
- AC5：注册表检测到同 id 冲突时拒绝后注册的包并输出明确错误信息。
- AC6：注册表对外暴露 `listAll()`、`findByName(name)`、`findByTag(tag)`、`search(keyword)` 四个 API。
#### FR-H04：按需注入（On-Demand Injection）
用户指定品牌名时，系统自动加载对应品牌包的双层内容：MD 注入 LLM prompt 传递设计意图，TS 约束激活用于生成后校验。
- AC1：用户在请求中提及品牌名（如"用 Apple 风格"）时，路由层通过注册表匹配对应品牌包，匹配失败时返回可用品牌列表供用户选择。
- AC2：匹配成功后，`design.md` 全文注入当前生成任务的 LLM prompt context，作为设计意图参考。
- AC3：匹配成功后，`index.ts` 中的 DesignSystemPackage 激活为当前任务的约束源，Domain G 的 constraint-checker、css-generator、constraint-injector 自动切换到该包。
- AC4：同一任务只能激活一个品牌包，用户指定多个品牌名时系统要求用户明确选择一个。
- AC5：未指定品牌名时，系统使用默认品牌包（可通过配置指定，初始默认为 dark-sci-fi）。
- AC6：品牌包切换不影响已生成的历史产物，只影响当前及后续生成任务。
#### FR-H05：内置品牌包（Built-in Brand Packages）
`npm install` 后自带全部 72 个品牌包（整合自 Open Design 生态），用户无需额外配置即可使用任意品牌风格。
- AC1：全部 72 个品牌包作为内置包随产品发布，覆盖 Apple、Stripe、Vercel、Linear、Notion、Figma、Tesla、Nike、Spotify、Supabase 等主流品牌，来源包括 Open Design 手工包（4 个）、bergside/awesome-design-skills 导入包（57 个）和 VoltAgent/awesome-design-md 导入包（70 个），去重后合计 72 个。
- AC2：每个内置包包含完整的三文件结构（`design.md` + `index.ts` + `reference.html`），通过 FR-H07 质量门禁全部检查项。
- AC3：内置包的 `tokens.colors` 至少包含 8 个命名色，`tokens.typography` 至少包含字体族和 5 级尺寸，`components` 至少包含 5 个组件定义。
- AC4：`npx claw-design init` 执行后，终端输出可用品牌包列表（含名称、分类标签和一句话描述），用户能立即看到全部 72 个可选风格。
- AC5：内置包的 `reference.html` 在浏览器中打开后能展示该品牌风格的完整视觉效果，作为生成基准。
- AC6：内置包随版本更新迭代，更新时保持向后兼容（不删除已有 token key，只新增或修改值）。上游 Open Design 包更新时，通过转换器重新同步。
#### FR-H06：社区包扩展（Community Package Extension）
支持用户自建品牌包并注册到系统中，格式与内置包完全一致。
- AC1：用户在 `src/design-system/packages/` 下创建符合三文件结构的目录后，系统自动识别并注册为可用品牌包。
- AC2：提供 `npx claw-design create-package <name>` 脚手架命令，生成品牌包目录骨架（含 `design.md` 模板、`index.ts` 类型骨架和空 `reference.html`）。
- AC3：社区包与内置包享有完全相同的能力：可被按需注入、可驱动约束校验、可出现在品牌列表中。
- AC4：社区包的 `index.ts` 必须通过 TypeScript 类型检查（符合 DesignSystemPackage 接口），类型不匹配时注册失败并输出具体字段错误。
- AC5：社区包可通过 npm 包形式分发，安装后自动被注册表发现（约定目录结构：`node_modules/<pkg>/design-system/packages/<name>/`）。
#### FR-H07：包质量门禁（Package Quality Gate）
注册新品牌包时自动执行质量校验，确保包的完整性和可用性达到最低标准。
- AC1：MD 结构校验——检查 `design.md` 是否包含 9 段标准结构中的至少 5 段必要段（Color Palette、Typography、Component Stylings、Layout Principles、Visual Theme），缺少必要段时报告具体缺失项。
- AC2：TS 字段覆盖率校验——检查 `index.ts` 导出的对象是否覆盖 DesignSystemPackage 接口的全部必填字段，字段覆盖率低于 80% 时标记为不合格。
- AC3：双层一致性校验——抽检 `design.md` 中声明的主色（前 3 个 hex 值）是否出现在 `index.ts` 的 `tokens.colors` 中，不一致时输出警告。
- AC4：reference.html 可渲染性校验——检查 `reference.html` 文件存在且非空，HTML 结构可被浏览器解析（无未闭合标签）。
- AC5：质量门禁结果分三级：通过（全部检查项合格）、警告（非必要项不合格，包可用但建议修复）、阻断（必要项不合格，包不可注册）。
- AC6：质量门禁在以下时机自动触发：注册表热重载扫描到新包时、`npx claw-design validate-package <name>` 手动触发时、CI/CD 流水线中作为发布前置检查时。
#### FR-H08：约束校验集成（Constraint Validation Integration）
生成完成后自动使用当前激活品牌包的 TS 约束做合规检查，复用 Domain G 已有的 constraint-checker 能力。
- AC1：生成产物经过 Domain G FR-G04 的约束注入后，再经过 FR-G05 的视觉验证，两层校验均使用当前激活品牌包的参数，品牌包切换后校验规则自动跟随切换。
- AC2：约束校验覆盖：颜色值必须来自当前包的 `tokens.colors`、字号必须来自 `tokens.typography.sizes`、间距必须来自 `tokens.spacing.values`、圆角必须来自 `tokens.radius`。
- AC3：校验失败时输出具体违规项（违规位置、实际值、期望范围），并触发自动重新生成（最多 3 次，复用 FR-G04 AC3 的重试机制）。
- AC4：内置包和社区包使用完全相同的校验流程，无差别对待。
- AC5：当品牌包的 `constraints` 字段定义了自定义校验函数时，除通用校验外额外执行自定义校验，自定义校验失败同样触发重新生成。
## 二、非功能需求（NFR）
### NFR-01：可靠性
- 相同输入在相同上下文下结构一致率 ≥ 95%。
- 单步骤失败时，系统在 5 秒内返回失败位置和原因。
- 交付前 100% 形成明确结论（成功/警告/失败），假交付率 = 0。
### NFR-02：性能
- 路由结论 ≤ 3 秒，澄清清单 ≤ 5 秒。
- 单页产物首稿 ≤ 30 秒，多页任务主骨架 ≤ 60 秒。
- 质量门禁首轮结论 ≤ 10 秒。
### NFR-03：可编辑性
- PPTX 导出文本可编辑率 ≥ 90%。
- 核心链路整页图片退化率 ≤ 5%。
- HTML 和 PPTX 结果均支持增量修改。
### NFR-04：视觉一致性
- 同任务交付物主题偏差项 ≤ 2 处。
- 变体探索限定在 Theme Pack 声明边界内。
- 跨格式核心视觉语言保持率 ≥ 90%。
### NFR-05：可扩展性
- 新 Skill 从合同声明到路由可用 ≤ 1 人天。
- 新增交付类型不修改已有路由/质量/导出代码。
- 增强包叠加不破坏开源核心功能。
### NFR-06：可移植性
- 核心运行零平台专属依赖。
- 交付结果在 ≥ 3 种主流浏览器/办公工具中可用。
- 独立分发包体积 ≤ 50 MB。
### NFR-07：可观测性
- 任务追踪覆盖率 100%（路由结论 + 生成结果 + 质量结果 + 交付清单）。
- 失败归因可定位到 7 段管线中的具体环节。
- 关键失败案例 48 小时内可复盘。
### NFR-08：可用性
- 非技术用户无指导完成首次任务成功率 ≥ 80%。
- 结果摘要零内部术语。
- 多产物任务用途说明覆盖率 100%。
### NFR-09：国际化
- 中日韩文本渲染准确率 ≥ 99%。
- 日期/数字/单位自动适配任务语境。
- 同一能力集支持 ≥ 3 种语言地区。
### NFR-10：无障碍
- 文本颜色对比度 ≥ 4.5:1（WCAG AA）。
- 图表和图解辅助说明覆盖率 100%。
- 演示型结果支持完整键盘导航。
### NFR-11：安全与内容边界
- 跨任务数据泄露事件 = 0。
- 上下文复用越界率 = 0。
- 交付结果零调试信息和过程痕迹。
### NFR-12：成本可控与外部依赖约束
- 文本 LLM 是唯一必要的 AI 依赖，用户只需配置文本 LLM 即可使用全部 11 个设计 Skill 的核心功能，零额外模态配置成本。
- 核心链路零外部图片 API 调用，CSS/SVG/代码渲染保证零门槛可用。
- 图像生成 API（DALL-E、Gemini 等）作为可选增强层接入，用户配置 API key 后自动启用，提升视觉表现力上限。
- 联产任务中间结构复用率 ≥ 70%。
- 开源能力覆盖 ≥ 80% 商用交付场景。
## 三、关键架构约束
### AC-01：路由优先架构
系统按意图路由 → 结构成稿 → 质量门禁 → 导出交付的顺序工作，后续阶段不能跳过前置判断直接出稿。
### AC-02：HTML 为主表达，核心交付物支持 HTML + PPTX
HTML 负责稳定展示、链接分享和后续复用，PPTX 负责编辑与企业协作，两者共同构成核心交付标准。
### AC-03：代码渲染为主，模态模型可选增强
核心产物采用文本、版式、SVG 和结构化图形表达，保证零门槛、零成本、可控可审查。CSS/SVG 渲染是完整的主渲染路径，产出成品级交付物。图像生成 API 作为可选 ImageProvider 接入，配置后自动增强视觉表现力上限，未配置时主渲染路径独立完成全部产出，无功能缺失。文本 LLM 是唯一必要的 AI 依赖；图像生成、视频生成、视频理解等模态模型均为可选增强项，不影响核心设计能力的完整性。
### AC-04：Theme Pack 先于生成
主题包和品牌上下文在生成前确定，生成过程必须服从其边界。
### AC-05：质量门禁先于交付
质量门禁是硬流程。阻断项未清除前，任务不能进入正式交付状态。
### AC-06：能力模块化
每类交付物作为独立 Skill 存在，同时接入统一主题、统一质量和统一导出规则。
### AC-07：独立产品边界与宿主适配分离
产品核心能力保持独立，宿主平台只负责接入、补问和分发，不吞掉核心产品边界。
## 四、竞品差异化与商业化路径
### 产品差异化
- 统一入口：用户面对的是任务入口，不需要自己拼工具链。
- 双结果交付：展示和编辑两种需求在同一任务里同时满足。
- 多产物联产：主交付物和附属传播物料共用一条信息主线。
- 主题化复用：品牌规则、模板骨架和组件资产可以持续积累。
- 质量先行：质量门禁直接影响交付动作，减少返工和失误外流。
- 开源独立：产品可独立运行，也适合接入更大的 Agent 工作流。
- 零模态配置成本：用户只需配置文本 LLM，无需额外配置图像生成、视频生成等模态模型，即可使用全部设计能力。所有 Skill 的渲染层基于 HTML/CSS/SVG/JS，开箱即用。
### 商业化判断
Claw Design 最强的市场切入点是“内容到多格式交付包”这条链路。企业真实需求集中在汇报、提案、图解、宣传和可编辑交付，长期价值来自复用和治理，不来自一次性出图。
- 第一层：开源核心能力，开放统一入口、核心交付类型、基础质量门禁和双结果链路。
- 第二层：增强主题与模板资产，提供行业模板包、品牌主题包、联产骨架包和更强的质量规则。
- 第三层：企业增强能力，提供团队品牌库、任务治理、扩展质量策略和更细的交付治理能力。
### 竞品判断
面向通用图片生成工具，Claw Design 在商业文档、图表、架构图、流程图和多格式交付方面更稳。面向单点设计技能工具，优势来自统一路由、统一主题、统一质量门禁和统一导出标准。面向封闭式宿主体验，走的是可开源、可扩展、可复用、可接入的路径。
